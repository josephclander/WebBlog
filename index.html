Some of the tasks were easier than others. However, I have always made a point of completing a task, passing the tests, and then researching to see how others completed it. This has helped tremendously through the earlier tasks FCC set and enabled me to have a better understanding of what makes better code. Some of your code may pass but is unnecessarily long, solved by brute force, has other bugs or is just not as elegant and readable as it could be.

The tasks have been great (albeit frustrating at times) at highlighting areas that you still need to work on. Some of the tasks made me consider the number of calculations the code may take and begin to research more on the big O. Adding Keyboard Accessibility

As I continue to see more and more articles on improving accessibility I realise that it is not until you start to test your site that you see truly how it feels to navigate with certain restrictions. And how annoying it gets when it just doesn't work!

Playing with the new faded overlay I added to the bottom of the articles for asthetic purposes, I decided to just use the tab key to navigate the page. Starting from the url, the tab key sent me down the page to the first instance where I have an anchor. Most of the articles are missed and because I added a hidden section that's only revealed on clicking a div element, you can't even open them to read.

Doing some research, I found that if I stick to the rules of semantic HTML and make my div element a button element, then the tab key will focus on it and simply using enter will expand the selection. I did have to add the style "border: none" to break the user styles again but this feels much better now.

This doesn't mean the problem is completely solved for keyboard access. Now as you navigate the page, for any articles that you don't open, you will still navigate those links unnecessarily and shift the text in the article down so you can't read the headline. However, an immediate fix isn't required as the tab goes to the expand button first before the links so it's most likely you didn't want to read it in the first place.

Monday 2nd July 2018

Book Review: UX for Beginners by Joel Marsh

There were many interesting things that came out of the book for me. I already appreciated that UX was not a means of making a beautiful design but the book had pushed to show that a great deal of thought on psychology goes into the way a UX designer creates a website. A lot of the approaches that caught my attention were looking at how to not just make the sites accessible and intuitive to the user but to make them “feel” like they were making the correct choices. As a web designer, I am to look at what the site’s goals are and what the user’s goals are. I need to make it such that when the user gets their goal, so does the website and it does need to be in that order.

It was interesting to read about the methods used to grab a user’s attention; gamification, colour schemes, lining up buttons along the “axes of interaction” (or not) and use of dummy prices to give the look of value to name but a few. It made you look at some of your most used sites in a new light as you can now critique them on how they attempt to keep your attention. It also highlighted to you how certain sites are trying to trick you. Hunting to find buttons that have negative actions for the site or the use of click-bait become even more obvious. The book doesn’t focus on exposing the sneaky sites, it gives a good exposure to accessibility features and how your design could be improved to help all too.

When trying to design, I have now learned that aiming to solve problems that you see in front of you is a better approach than simply brain storming good ideas. This was quite a standout point for me as you do imagine designers just sitting around trying to come up with cool ideas. The approach means that as you discover what the problems or limitations are, you will be able to make sure all are taken note of and that fewer problems are created if at all. This resonated with me from the use of Linear Programming in Mathematics. This looks to form equations of all the issues and graph them. You then look for the areas of the graph that area valid and select the ones that give you the best result from that list.

The methods used to test your site for effectiveness also have a very mathematical feel to them. I have spent a long time in my profession being told about how the data makes the point the tester wants it too, when in reality it hasn’t been tested fairly. The use of surveys appears almost pointless without leading the users except for a few circumstances where minimal, factual information is required. A/B testing appeals to my science background as they try to use large samples and change only one element. This is what makes a test fair. It also hit home about how people’s memories do not always tell the truth. This applies to both the user and the tester. Use of film to see user’s reactions alongside facts such as the ones you can use with Google’s tools will give a much more accurate picture of what is going on. Again, these must be used as factually as possible and not interpreted.

There were many more points made in this book worth reading. I have only scratched the surface.

Saturday 23rd June 2018

User Agent Styles - Hidden Issues

Oh my god. How hard is it to style a button? I couldn’t believe that no CSS styling would work on the submit button on my form. Every site said that it was easy but I could only get width to make ANY difference. With further research, it appears that the user agent styles were acting strangely for me. To get any of my styles to work I had to change a border that I didn’t need to change. Why is that? Luckily I stumbled across this site on overriding default button styles, otherwise I would have assumed that it couldn’t be done, at least not by me.

And release!

NOTE: User Agent Styles are the CSS styles that every web browser uses out of the box to style HTML. This means that if you didn’t create any CSS at all to link to your document, that it would still be legible, if still exceedingly dull and awkward.

While considering the topic, it reminded me that other issues I had were also due to factory settings. For example, it took me quite a while to discover why “margin: 0 auto” on my body tag, was not centering my container div. On closer inspection the body does not automatically cover 100% of the window and I had assumed it did. It’s completely white and you can’t see it so why not? It turned out my div was centred in the div! So “width: 100%” on the body fixed it. Although I didn’t use the following technique for this issue, I had seen that you can add a single pixel border to objects so that you can see them and understand what they are doing until you are ready to remove them.

If these styles are becoming a menace and you want full control, there are CSS stylesheets out there that will reset everything to zero such as Eric Meyer's Reset Tool. There are ups and downs to this. Yes, there will be fewer styles that you are trying to override as you get to build them from scratch but it also means it takes a long time. For a newbie like me, it’s worth looking at the final product to see if you’ve kept the code DRY (Don’t Repeat Yourself) as I may move things just on a visual basis and at the end realise that every paragraph tag has different padding when it truly wasn’t necessary.
